# Электронная очередь приёмной комиссии — дизайн

**Дата:** 2026-06-08
**Статус:** утверждён, готов к плану реализации

## 1. Цель

Онлайн-система электронной очереди для приёмной комиссии колледжа. Абитуриенты
встают в очередь с телефона (по ссылке/QR), видят свою позицию в реальном времени
и сигнал о вызове. Сотрудники комиссии (операторы) вызывают абитуриентов по своим
направлениям, общее табло в холле показывает вызываемые номера.

## 2. Ключевые решения

- **Формат:** онлайн, абитуриент заходит с телефона анонимно (без регистрации).
- **Структура:** несколько очередей по направлениям/специальностям; у направления
  может быть несколько окон.
- **Хостинг:** развёртывание в интернете (Vercel).
- **Уведомления:** страница абитуриента обновляется в реальном времени, при вызове —
  звук и вибрация в браузере. Без SMS и push.
- **Стек:** Next.js (App Router) на Vercel + Supabase (Postgres + Realtime + Auth).

## 3. Архитектура

- **Next.js (App Router)** — все экраны как страницы одного приложения, серверные
  маршруты для мутаций.
- **Supabase Postgres** — единый источник правды.
- **Supabase Realtime** — все экраны подписаны на таблицу `tickets`; изменение статуса
  мгновенно расходится по подписчикам.
- **Supabase Auth** — вход только для операторов и админа. Абитуриент анонимен.

### Маршруты

| Путь | Кто | Назначение |
|------|-----|-----------|
| `/` | Абитуриент | Выбрать направление, встать в очередь |
| `/ticket/[id]` | Абитуриент | Своя позиция, статус, звук при вызове |
| `/board` | Табло | Большой экран: вызываемые номера и окна |
| `/operator` | Оператор | Своя очередь, кнопки управления |
| `/admin` | Админ | Направления, окна, статистика, сброс дня |

## 4. Модель данных

**`services`** — направления/специальности
- `id`, `name` (напр. «Подача документов»), `prefix` (буква талона, напр. «A»),
  `is_active`

**`counters`** — окна
- `id`, `name` (напр. «Окно 1»), `service_id` (обслуживаемое направление), `is_open`

**`tickets`** — талоны
- `id`, `service_id`, `number` (порядковый за день, напр. A-014)
- `status`: `waiting` / `called` / `serving` / `done` / `no_show`
- `counter_id` (кто вызвал), `recall_count` (число повторных вызовов)
- `created_at`, `called_at`, `finished_at`

**`operators`** — связь пользователя Auth с ролью/окном
- `user_id`, `role` (`operator` / `admin`), `counter_id`

**Нумерация:** при создании талона `number` = `prefix` + следующий порядковый номер
внутри направления за сегодня (A-001, A-002, …). Сброс — раз в день или вручную в
админке.

## 5. Экраны

### Абитуриент `/`
Список активных направлений крупными кнопками. Нажатие создаёт талон и редиректит на
`/ticket/[id]`.

### Талон `/ticket/[id]`
- Крупно свой номер (A-015).
- «Перед вами: N человек» + примерное время ожидания (N × средняя длительность).
- Статус: ⏳ Ожидание → 🔔 «Вас вызывают! Окно 2» (звук + вибрация) → ✅ Завершено.
- Realtime-подписка на `tickets` своего направления. Кнопка «Покинуть очередь».
- Номер/ссылка сохраняются в localStorage для возврата после закрытия вкладки.

### Табло `/board`
Полноэкранное для ТВ. Таблица последних вызовов (A-014 → Окно 1). Свежий вызов
подсвечивается + звуковой сигнал. Realtime.

### Оператор `/operator`
После входа видит своё окно и направление, текущего обслуживаемого и число ожидающих.
Кнопки:
- **«Вызвать следующего»** — старейший `waiting` талон направления → `called`,
  проставляет `counter_id` и время.
- **«Повторный вызов»** — повторно зажигает табло и сигнал абитуриенту, не меняя
  позицию; увеличивает `recall_count`, обновляет `called_at`.
- **«Завершить»** — текущий → `done`.
- **«Не явился»** — текущий → `no_show`, можно сразу вызвать следующего.

### Админ `/admin`
- CRUD направлений и окон, открыть/закрыть окно.
- Статистика за день: обслужено, среднее время ожидания/обслуживания, число «не явились».
- **«Сбросить день»** — архивирует талоны и обнуляет нумерацию (с подтверждением).

## 6. Поток в реальном времени

Оператор жмёт «Вызвать» → `UPDATE tickets` в Postgres → Supabase Realtime рассылает
событие → одновременно обновляются страница абитуриента (звук при вызове), табло
(новая строка), экран оператора. Единый источник правды — таблица `tickets`, все
экраны слушают её.

## 7. Безопасность (Supabase RLS)

- **Абитуриент (аноним):** создать талон; читать талоны своего направления для расчёта
  позиции. Не может менять чужие талоны или вызывать.
- **Оператор:** менять статусы талонов только своего направления.
- **Админ:** всё, плюс управление `services` / `counters`.
- Правила задаются на уровне БД (RLS), их нельзя обойти из браузера.

## 8. Граничные случаи

- **Двойной вызов** двумя операторами одного направления → атомарный апдейт по условию
  `status='waiting'`; задвоения нет.
- **Пустая очередь** → «Вызвать» сообщает «никто не ждёт».
- **Абитуриент закрыл вкладку** → талон жив; возврат по сохранённой ссылке восстанавливает
  позицию.
- **Вызвали отсутствующего** → по таймауту оператор отмечает «не явился» и идёт дальше.
- **Сброс дня при активных талонах** → подтверждение + архивация (статистика не теряется).
- **Потеря Realtime-соединения** → клиент переподключается и подтягивает текущее состояние
  fallback-запросом.

## 9. Тестирование

- **Юнит-тесты:** выдача следующего номера, выбор следующего ожидающего, атомарность
  вызова, подсчёт «перед вами».
- **Интеграционные тесты:** API-маршруты (создать талон, вызвать, завершить) против
  тестовой схемы Supabase.
- **Ручной сценарий** перед комиссией: табло + 2 окна + телефон, полный цикл.
- **Подход:** TDD для ключевой логики очереди — сначала тест, затем реализация.
